## [2026-03-22 00:22:00] [Session ID: codex-20260321-234700] 主题: PyPI NVIDIA wheels 对原生链接器并不总是“即装即链”

### 发现来源
- 在修复 `install-tinycudann` 时,先后遇到:
  - `nvrtc.h` / `cusparse.h` / `cublas_v2.h` 缺失
  - `cannot find -lnvrtc`

### 核心问题
- PyTorch 相关环境经常把 CUDA 生态拆成两层:
  - 系统 CUDA 提供 `nvcc`
  - PyPI / pixi 的 `nvidia-*` wheel 提供头文件与 `.so.12`
- 但这些 wheel 的 `lib` 目录里往往只有 `libfoo.so.12`,没有 `libfoo.so`。
- 编译器看到头文件还不够,链接器处理 `-lfoo` 时仍会失败。

### 为什么重要
- 这种失败表面上很像“库没装”或“CUDA 版本不兼容”,实际是文件命名和链接器约定不匹配。
- 如果不先识别这一层,很容易陷入一轮又一轮“继续补 include 路径”的假修复。

### 当前结论
- 当前已知事实:
  - 对这类混合环境,需要同时处理:
    - include 路径
    - lib 路径
    - 无版本 `.so` 链接别名
    - 运行时 `rpath`
- 仍未确认的部分:
  - 仓库里未来是否还有别的 CUDA 扩展也会撞上同一类问题

### 后续讨论入口
- 以后如果再有自定义 CUDA 扩展在 `pixi` / PyPI NVIDIA wheel 环境里报 `cannot find -lxxx`,先检查库目录里是不是只有 `.so.<version>`。

## [2026-03-22 00:16:43] [Session ID: 019d0ead-8f40-77b2-b6a3-2ed88d658c78] 主题: GPU 子进程评估前,父训练进程必须先主动释放 CUDA 状态

### 发现来源
- 在 `train_gs` 完成后自动拉起 `eval_gs` 时,父进程训练本体已经结束,但子进程依然发生 CUDA OOM。

### 核心问题
- “训练结束” 不等于 “父进程显存已经释放”。
- 只要 `model`、`optimizer`、`gt_images`、`intrinsics_list` 等大对象还在作用域里,Python 进程就会继续占着显存。
- 此时再起一个 GPU 子进程,就是两个进程同时争同一块显存。

### 为什么重要
- 这类问题很隐蔽。表面上看像是“评估太重”或“显卡不够”,实际是父进程生命周期管理没收尾。
- 如果不把它沉淀下来,以后任何“训练后自动评估 / 自动导出 / 自动渲染”的子进程链路都可能再踩一次。

### 当前结论
- 当前已知事实:
  - 手动单独跑 `eval_gs` 可以成功
  - 在 `train_gs` 内 auto eval 前释放训练期大对象后,入口级 smoke 的 auto eval 错误消失
- 仍未确认的部分:
  - 其他 GPU-heavy 子命令链路是否也需要同样的父进程显存收尾策略

### 后续讨论入口
- 以后任何 GPU 主流程如果要在尾部再拉 GPU 子进程,优先先查: 父进程是不是已经 `del` 大对象并 `empty_cache()`。
